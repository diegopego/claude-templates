# Decision log

One short entry per architectural or directional decision: what, why, strongest rejected alternative. Reversals are new entries pointing at the old one.

## 2026-07-10 — This repo is a meta-project and dogfoods its own charters

All project knowledge lives versioned in-repo (`.claude/memory/`), with roadmap, decision log, and audience changelogs — the same practices the charters prescribe. The agent's global memory is forbidden for project facts. Rejected: relying on the agent's global memory (breaks the "fresh clone resumes work" property the charters demand).

## 2026-07-10 — Deliverables stay under `templates/`, made inert by rules, not by name

Kept the existing directory name; the anti-contamination guarantees come from CLAUDE.md rules (nothing under `templates/` governs this repo; no `CLAUDE.md` outside the root; no deliverables in `.claude/`). Rejected: renaming to `catalog/` or `dist/` — churn without additional safety.

## 2026-07-10 — Naming: SCREAMING_SNAKE for deliverables, kebab-case for meta-project files

The visual contrast marks the product/backstage boundary. A future CLAUDE.md template will be named `CLAUDE_MD.template.md` so Claude Code never auto-loads it. Rejected: one uniform convention (loses the boundary signal).

## 2026-07-10 — English is the source of truth; pt-BR translations are generated, committed, hook-enforced

Translations live at `templates/i18n/pt-BR/` mirroring the English tree, regenerated by the `translate-templates` skill; a PreToolUse hook blocks `git commit` when an English template is staged without its counterpart, or when templates/ideas change without a changelog update. Rejected: translating on demand only (goes stale silently) and a pure git pre-commit hook (a shell hook can detect staleness but cannot translate — that needs the LLM).

## 2026-07-10 — Products are multi-tenant from v1 (`product-not-bespoke`)

Every project the templates govern is a product for an audience by default (assumption declared in the scope proposal when unstated), and the product is born multi-tenant: the interlocutor's organization is tenant #1. Owner decided this **against the agent's recommendation** of single-tenant-but-configurable — rationale: retrofitting tenancy is among the most expensive migrations, so it counts as a today-requirement, a deliberate exception to build-for-today. Legacy extraction classifies each rule (domain | instance-config | workaround) and consolidates instance values into tenant #1's configuration profile. Rejected: single-tenant v1 with parameterized config (cheaper, but pays the migration later); blocking setup question with no default (friction on every project).

## 2026-07-10 — Work directly on `master`; no feature branches

The owner works this repo directly on `master` — it is a simple, single-owner meta-project and per-feature branches add ceremony without benefit. Commit straight to `master` (still only with explicit per-instance authorization); do not propose or create branches. Rejected: feature-branch-per-change (the default git hygiene, but unwanted friction here).

## 2026-07-10 — The Q&A-round method is defined once in the charter core, and the meta-project dogfoods it

The owner reported not understanding the "question-and-answer flow." Diagnosis: "Q&A round" was a term of art used ~8× across the charters (Align, "Ask, don't infer," Prototype loop, traceability, conflict arbitration) but **never defined**; the *how* was scattered — "options + a recommendation" lived only in `CLAUDE.md` (graduation-only), "scripted-then-adaptive" only in the `adaptive-setup-questions` draft (Setup-only). Fix: a single **Working through questions** section in `CHARTER_CORE.md` defines the round once (batched · options + recommendation · scripted-baseline-then-adaptive · answers become artifacts with traceability · closes at the phase's Exit); the phase list points to it. `CLAUDE.md`'s graduation rule now runs the *same* method — processing an inbox entry is a Q&A round whose resolutions are recorded in `decisions.md` before the spec is drafted (dogfooding). `adaptive-setup-questions.md` is marked **incorporated**, broadened from Setup-only to the general method. Composed charters regenerated (new section 3; later sections renumbered). Owner's steer: the round's home is the moment of processing an inbox item — ask up front, store well-made decisions. Rejected: leaving "Q&A round" undefined and only clarifying `CLAUDE.md` (misses the templates, where the term is used most); keeping scripted-then-adaptive scoped to Setup (it is how every round works, not a Setup quirk).

## 2026-07-10 — Landing skin designed once at Setup; Setup questions are scripted-then-adaptive

Two inbox entries (site-creation pipeline; scripted vs. adaptive questions) resolved with the owner. (1) The "site" is the **landing** (= living-doc rendered), not a separate site or the product frontend — so the site-creation pipeline is the landing's **visual identity**, decided once at Setup via a design interview (theme/palette/reference-site/audience/stack, Claude Design as default tool). This gives the landing two layers: **skin** (designed once, fixed so it doesn't drift) and **content** (the living-doc, regenerated forever). Folded into `landing-publishing.md` rather than a new spec. (2) The scripted-then-adaptive questioning idea is a **general Setup mechanism** (own spec `adaptive-setup-questions.md`), not scoped to the site: a fixed baseline guarantees coverage, Claude-generated follow-ups add context depth. Rejected: treating "site" as the product frontend (out of landing's scope — that's Build/charter); a dedicated site-template spec (skin fits inside `landing-publishing`); scoping adaptive questions to the site only (the mechanism serves all of Setup).

## 2026-07-10 — A richer Setup, and a changelog → living-doc → landing pipeline

Consolidated an inbox cluster (graduated as separate specs) into one model:
- **Setup (step 0) also scaffolds** — asks stack/language/target-audience/information-sources and generates a complete initial project setup (`setup-scaffolds-project`, draft).
- **Two curated changelogs, not multi-audience** (owner narrowed it): a **technical** changelog (git history is its raw record underneath) **and** a **target-audience** changelog. Owner chose to keep both rather than let git history replace the technical one (`audience-aware-changelogs`, revised).
- **Pipeline:** the *target-audience changelog → living-product-doc → single landing page* (for the target audience; not per-audience). The living-doc update consumes the target-audience changelog entry rather than the raw diff — this reconciles "landing derives from the living-doc" (prior decision) with "the changelog drives the landing" (transitively). Rewiring `update-product-doc` is pending incorporation. The agent picks information sources (git log, tests, code) and asks what it needs.
- **Publishing:** Artifact + GitHub Pages, with Claude Design available for visual prototyping (`landing-publishing`, draft).

Rejected: multi-audience changelogs (narrowed to two); git-history-only with no curated technical changelog (owner kept both); one landing per audience / per-audience sections (moot once not multi-audience — single landing for the target audience).

## 2026-07-10 — Living landing page (root README.md), maintained by a pre-commit skill

Added a living product document as the root `README.md`: the current-state, adopter-facing source of truth, kept current by the `update-product-doc` skill, which reads the staged diff before a commit and translates deliverable changes into user terms. Hook rule 4 blocks a commit that changes an adopter-facing deliverable (composed charters, requirements, guides, catalog) without staging `README.md`. Decisions: it is a **distinct** artifact from the changelog (current state vs. history) and from `audience-aware-changelogs` (kept separate, not merged); it lives at the **repo root, not under `templates/`** (a landing page about our product is not a deliverable — anti-contamination); it is English (meta-project artifact; the pt-BR exception covers only `templates/`). A rendered/published landing page (Artifact) may be derived from it later — the markdown is canonical. Rejected: merging manual + changelog into one system; evolving `templates/README.md` (the catalog) into the landing page (conflates catalog with product front page).

## 2026-07-10 — Charters get a Setup (step 0) that confirms all Project Parameters

## 2026-07-10 — Adoption merge resolves conflicts by always asking (no default winner)

Authored `templates/guides/GUIDE_ADOPTION.md`. When an existing project instruction conflicts with a template practice, the merge stops and asks the owner with a recommendation — neither existing-wins nor template-wins is a default. Owner chose this over the recommended existing-wins-and-surface: safest for a live system, and consistent with the charters' Q&A alignment. Also settled: the guide covers projects with no instructions (inventory finds nothing → charter selection + memory seeding, no merge); code/infra practices (e.g. portable-appliance) always defer to the roadmap, never applied during the merge.

## 2026-07-10 — Adopting the templates into an existing project is its own `GUIDE_` deliverable

Bringing these practices into a live, working repo (with its own instructions) is a third axis, distinct from both charters — the project keeps running; only its instruction/practice layer is adopted via a non-destructive merge. Handled as a new standalone deliverable type, `GUIDE_ADOPTION.md` (new `GUIDE_` prefix, home `templates/guides/`). Spec drafted at `ideas/adopt-into-existing-project.md`; the merge's conflict default (existing-instructions-win-and-surface) is proposed, pending the draft→agreed gate. Rejected: a section inside both charters (buries a cross-cutting concern, duplicates text); a skill-first approach (premature before the process is written — the guide can later be automated into the first `templates/skills/` entry).

## 2026-07-10 — The idea-inbox practice ships in the templates (core section + `graduate-idea` skill)

The meta-project's own inbox → graduation → spec loop graduates into a product feature so adopters get it too. Resolved in a Q&A round on the inbox entry:

- **Home: `CHARTER_CORE`**, a new *Idea inbox* section beside *Roadmap & decision log* — the practice is transversal (greenfield and legacy alike), like memory/roadmap/decisions.
- **Scope: inbox + graduation ritual, reusing the existing `spec_term`** — an `ideas/inbox.md` capture file plus a graduation ritual (Q&A round → write-up in the charter's ordinary golden-source specs + a roadmap entry). No second spec format is introduced; in a real project a graduated spec becomes built software, not template text.
- **Ship a skill now, not just prose** (owner overrode the recommendation to defer to Milestone 4): a `graduate-idea` skill under `templates/skills/` — the first embeddable skill template.
- **Skill covers graduation only** — capture is a trivial one-line append to the inbox, documented in the charter prose; the skill concentrates on the valuable part (the Q&A round → a well-formed spec).
- **This skill defines the skill-template convention** — establishing the embeddable-skill format (`templates/skills/<name>/SKILL.md`, a "copy me into `.claude/skills/`" adoption note, kebab-case naming), closing the first half of Milestone 4.

Rejected: a pluggable module or a standalone `GUIDE_` (the practice is universal — it belongs in the core, not opt-in or out-of-charter); porting the full meta-project machinery incl. the detailed spec format and the changelog rule (redundant with the existing `spec_term`, and a real project's specs are built, not published); capture-only or a single do-everything skill (capture is trivial; two triggers in one skill hurt auto-selection); deferring the skill-template format to a later round (owner wanted the reference example built now).

## 2026-07-10 — Graduated the Setup→changelog→landing inbox cluster (2 Q&A rounds)

The owner authorized graduating the five newest inbox entries, which turned out to be refinements of the pending Milestone-3 draft cluster. Resolved across two Q&A rounds and folded into the idea-specs (moved to `agreed`); template-text incorporation is a distinct later step. Decisions:

- **Spec-driven work becomes a new `CHARTER_CORE` section** (inbox #3). Every non-trivial task is turned into a mini-spec before implementation; gaps that would prevent an ideal spec trigger an immediate Q&A round (reusing *Working through questions* and the existing `spec_term`). It is the same machine as the *Idea inbox*, extended from "ideas" to "tasks" — transversal (greenfield + legacy), so it lives in the core, not a module. New idea-spec: `spec-driven-work.md`. Rejected: routing every task through the idea-inbox (conflates "task to implement" with "idea to mature"); a pluggable module (the discipline is universal).
- **The update skill works by delta from a versioned last-processed-commit marker** (inbox #1), not the pre-commit staged diff. The marker (a field under `.claude/memory/`) lets the skill run on demand over an interval and advance the marker when it runs, decoupling doc/landing/changelog updates from the commit moment. Rejected: keeping the staged-diff model (tied to the commit, no interval); deriving the delta from git history alone (fragile under history rewrites / unrelated artifact edits).
- **Every landing update passes an approval loop** (inbox #2): textual statement of intent → rendered landing preview → owner approval → publish. Publishing is never automatic — it is outward-facing, and the owner asked to approve. Rejected: approving only skin/structural changes (owner's words were explicit about every update); approval only on first publish + rebrand.
- **The Setup CLI is specified now, at full-lifecycle scope** (inbox #4) — against the agent's recommendation to defer it as a future direction. The CLI is the project's single door (setup, adopt, doc/landing update, graduate-idea, …), styled after `claude`. As a meta-project we ship the *specification*, not the implementation; new idea-spec `lifecycle-cli.md`, later graduating to a `REQUIREMENT_`. Rejected: deferring to the roadmap (owner wanted it specified now); scoping the CLI to Setup+scaffold only (owner chose the comprehensive single-door tool); a thin wrapper over Claude/skills (no scaffold logic of its own).
- **Tech-stack choices are the developer's, made at Setup** (inbox #5). The charter's biased picks (strict TypeScript, functional-core / imperative-shell) become *recommended defaults — stated and overridable*, never silent assumptions; consistent with the existing "never assume a parameter" Setup rule. Folded into `setup-scaffolds-project.md`.
- **Two changelogs are a core practice, with split cadence** (audience-aware-changelogs open questions). A `CHARTER_CORE` section, not a pluggable module — every project wants a technical + a target-audience changelog, universal like memory/roadmap; the target audience is a Setup parameter. Cadence differs: the **technical** changelog is updated **per commit** (git history is its raw record underneath); the **target-audience** changelog is **curated per significant change**, grouping commits to avoid noise for the end user. Rejected: a `MODULE_DOCUMENTATION_BY_AUDIENCE` (not opt-in — universal); both-per-commit (noisy for the audience view); both-per-milestone (technical loses fine granularity).

Also softened *Stack philosophy*'s opener to "a recommended default the developer confirms or replaces at Setup." Placement: beside *Working through questions* (both are Method-elaboration sections) rather than folding all detail into the terse phase bullet. Numbering shifts +1 after the new section.
- **New requirement, `REQUIREMENT_PROJECT_CLI.md`.** A `claude`-styled, interactive, single-door CLI over the four lifecycle commands (`setup`/`adopt`/`update-docs`/`graduate-idea`), specified as a thin orchestrator.   - Also settled inline (not asked, adopted from existing decisions): CLI config (marker, skin tokens, stack choices) is **versioned in-repo under `.claude/`** (consistent with in-repo memory); distribution/runtime and exact flags are **non-goals** (implementation concerns).
- Catalog (`templates/README.md`) gained the new requirement. `ideas/setup-scaffolds-project.md` and `ideas/lifecycle-cli.md` marked **incorporated**; roadmap updated. Ran the full ritual: assemble → landing → translate → changelogs. The rendered **Artifact** landing was **not** republished — held for owner approval per the landing approval loop.
