One short entry per architectural or directional decision: what, why, strongest rejected alternative. Reversals are new entries pointing at the old one.

## 2026-07-11 — Landing moves to GitHub Pages; the rendered HTML is now versioned

Owner activated the parked GitHub Pages item right after the rewrite commit: the public landing is now `docs/index.html` on `master`, served by GitHub Pages at `https://diegopego.github.io/claude-templates/` (repo is public; `.nojekyll` alongside). ## 2026-07-11 — Rewrite from scratch in ideal form (owner accepted the risk)

After an honest assessment concluded the template content was strong but the meta-machinery outweighed the product (zero adopters after 4 milestones), the owner ordered a full rewrite — "reescreva do zero da forma ideal. eu assumo o risco." — and settled the outcome-changing questions in one Q&A round. Everything was re-authored; git history preserved. The decisions, each reversing or reshaping an earlier entry:

- **pt-BR translations dropped** (`templates/i18n/`, `translate-templates` skill, hook rule). Reverses *English is the source of truth; pt-BR translations are generated…* (2026-07-10): the pipeline doubled every diff for zero known consumers — the same build-for-today logic already applied to `translated-templates` on 2026-07-11. Translations return when an adopter asks.
- **One changelog instead of two.** Root `CHANGELOG.md`, curated for adopters per significant change; git history is the technical record. Reverses the audience-changelogs practice *for this repo* (the practice itself lives on in `MODULE_LIVING_DOCS`): the maintainer changelog had grown larger than any charter while duplicating `git log`.
- **Landing machinery replaced by a ritual.** The `update-product-doc` skill and `last-processed-commit` marker are gone. Per-commit freshness is kept — the owner's explicit constraint — by the hook (adopter-facing change ⇒ root `README.md` staged in the same commit; the editing agent knows the delta, no marker needed) plus a CLAUDE.md ritual line: after a commit touching README, offer to republish the Artifact (same URL), always with owner approval. Rejected: keeping the marker+skill pipeline (machinery for a delta the session already knows).
- **`REQUIREMENT_PROJECT_CLI.md` parked** back to `ideas/lifecycle-cli.md` (`deferred`), its Q&A resolutions preserved there. A normative spec for software no adopter asked to build was speculation; it reactivates when a real project wants the tool. Charter references to the CLI removed.
- **Charter core slimmed to 12/13 sections; opinionated practices became opt-in modules.** The Q&A-round method is defined once inside Method; *Idea inbox* + *Spec-driven work* merged into *Ideas & specs*; memory/roadmap/decisions merged; Setup keeps parameter confirmation + minimum runnable scaffolding. New add-on modules: `MODULE_PRODUCT_AUDIENCE` and `MODULE_LIVING_DOCS`; shipped composed charters are minimal. **This reverses *Products are multi-tenant from v1* (2026-07-10)** as a universal default — the practice's full text lives in `MODULE_PRODUCT_AUDIENCE`, applied when a project is a product for an audience, not imposed on every internal tool. The extraction charter keeps a tenancy-neutral *Classify on extraction* rule (domain | instance-config | workaround) so the valuable half survives in the minimal charter.
- **`templates/README.md` (catalog) merged into the root `README.md`** — one adopter-facing page instead of a landing that half-duplicated a catalog. Reverses the "distinct catalog" split (2026-07-10 living-landing decision) now that the tree is small enough for one page. Hook simplified accordingly (README required whenever anything under `templates/` changes — also closes the recorded rule-4 gap for `templates/skills/`).
- **Incorporated `ideas/` specs deleted** (10 files; git history keeps them) — a spec whose content lives in template text is a second copy waiting to drift. Milestones 1–4 moved to `roadmap-archive.md` (the charters' own strategic-archiving practice, finally applied here).
- **New direction: validation before growth.** The roadmap's next milestone is adopting the templates in a real project and iterating from real friction — no further meta-refinement until then.

## 2026-07-10 — Products are multi-tenant from v1 (`product-not-bespoke`)

Every project the templates govern is a product for an audience by default (assumption declared in the scope proposal when unstated), and the product is born multi-tenant: the interlocutor's organization is tenant #1. Owner decided this **against the agent's recommendation** of single-tenant-but-configurable — rationale: retrofitting tenancy is among the most expensive migrations, so it counts as a today-requirement, a deliberate exception to build-for-today. Legacy extraction classifies each rule (domain | instance-config | workaround) and consolidates instance values into tenant #1's configuration profile. Rejected: single-tenant v1 with parameterized config (cheaper, but pays the migration later); blocking setup question with no default (friction on every project).

The owner works this repo directly on `master` — it is a simple, single-owner meta-project and per-feature branches add ceremony without benefit. Commit straight to `master` (still only with explicit per-instance authorization); do not propose or create branches. Rejected: feature-branch-per-change (the default git hygiene, but unwanted friction here).
