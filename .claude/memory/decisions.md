# Decision log

One short entry per architectural or directional decision: what, why, strongest rejected alternative. Reversals are new entries pointing at the old one.

## 2026-07-11 — `make adopt` installs the permanent charter-prescribed skills, not just the one-shot merge tool

Inspecting the first real adoption (orderboard) showed its `.claude/skills/` empty — no `graduate-idea` — even though its adopted `CLAUDE.md` prescribes the idea-inbox/graduation practice `graduate-idea` automates. Root cause: `make adopt` installed only `adopt-template`, which the teardown removes, so an adopted project was left with **zero skills** after the merge. Fix: the adopt path now also installs `graduate-idea` (permanent, core-prescribed) and seeds an empty `ideas/inbox.md` if absent — additive and non-destructive (both skip when present). The permanent/one-shot split is made explicit in `GUIDE_ADOPTION` and the `adopt-template` teardown step: `graduate-idea` (and `update-living-docs` when the living-docs module is adopted) stay; only `adopt-template` and the kit are one-shot. Rejected: leaving the adopter to copy the skill by hand (the gap the owner hit); installing `graduate-idea` only via the merge (more indirection than an unconditional core practice warrants — the installer already installs `adopt-template` directly). Orderboard was fixed immediately by copying `graduate-idea` into its `.claude/skills/`.

## 2026-07-11 — Reactivate the living-docs pipeline as the `update-living-docs` skill (rebuild + update)

The first real adoption (orderboard) surfaced that the `change → audience changelog → living product doc → landing` pipeline existed only as prose in `MODULE_LIVING_DOCS` — no automation. The rewrite had deliberately removed the `update-product-doc` skill and parked the CLI under "zero adopters"; a real adopter wanting the pipeline is exactly the reactivation trigger those decisions named. Owner Q&A settled it, both as recommended: (1) build a **new embeddable skill** (not reactivate the full parked CLI, not enrich prose only) — its bones recovered from the removed `update-product-doc` (git history) and generalized for any adopter; (2) **couple it to `MODULE_LIVING_DOCS`** — delivered only when the module is adopted (the installer installs it when `MODULES` includes `living-docs`; the adopt kit now carries the module source + the skill so the merge installs it if the module is taken), not shipped standalone like `graduate-idea`/`adopt-template`.

A mid-build owner question — "can it rebuild, or do I need a separate force-recreate command, for a large in-progress project like orderboard?" — shaped the design: **one skill, two modes** — `rebuild` (full regeneration from current state: reconcile an existing changelog → technical, create and baseline-seed the audience changelog; the path for a large adopter or a forced re-creation) and `update` (incremental single-change fold) — rather than a separate command that would duplicate the pipeline and drift. Rebuild seeds the audience changelog with **one current-state baseline entry** instead of fabricating a low-fidelity per-commit past. Module and skill mirror each other; `README`/landing/`CHANGELOG` updated; `assemble` is a no-op (the module is in no composed charter). Rejected: reactivating `REQUIREMENT_PROJECT_CLI` now (far larger — the owner wanted the pipeline, not a CLI); a standalone always-shipped skill (pushes landing machinery onto projects that opted out of the practice); a separate rebuild command (pipeline duplication → drift).

## 2026-07-11 — Self-adoption force-refreshes its skill working copies; never-overwrite stays the default everywhere else

Second Milestone-6 friction (same orderboard session): after editing a skill source, CLAUDE.md's refresh path — "re-run `make adopt DEST=.`" — silently no-opped, because `install_file` skips any destination that exists (the never-overwrite invariant that protects adopter files). The self-adopted working copies went stale with no sanctioned way to refresh them short of a manual `rm`. Fix: `install_file` gains an opt-in `force` third argument; the self-adoption branch passes it (`install_skill … force`), so re-running refreshes both working copies from source and reports `refreshed`. This makes the already-documented CLAUDE.md path *true* rather than editing the doc around a papercut. Scoped to self-adoption only — `make new` and foreign `make adopt` still never overwrite, so the adopter-facing promise ("nothing is ever overwritten") is unchanged; it is safe here because working copies are generated, never hand-authored. Verified by corrupting a working copy and confirming the re-run overwrites it back to source. Rejected: a global `FORCE=` flag (more surface for a case self-adoption always wants anyway); editing only the CLAUDE.md wording to say "rm then re-run" (documents the papercut instead of removing it).

## 2026-07-11 — Adoption tears down its delivery kit; kept deliverables get a single versioned home under `.claude/`

The first real adoption (orderboard) surfaced a gap: `make adopt` deposits the full template kit in `agent/`, but neither `GUIDE_ADOPTION.md` nor the `adopt-template` skill said what becomes of it — so a kept `REQUIREMENT_PORTABLE_APPLIANCE` ended up duplicated between `agent/` and the adopter's own versioned copy, inviting the exact drift the adopter's single-owner rule forbids. Recorded as Milestone-6 friction, then fixed **directly in template text** (a spec whose content lives in the templates would be a second copy waiting to drift). An owner Q&A settled two choices, both as recommended:

- **The guide/skill own an explicit final teardown.** New step 6 *Tear down the delivery kit* — migrate each kept deliverable to a versioned home, then remove the install folder (`agent/`, `PREFIX`-overridable) and the one-shot skill; *Prove by functioning* renumbers to 7, and *Produce the output* gains a migration bullet. The installer stays a dumb deterministic file-installer (the same-day installer-boundary decision): "which keep goes where" is a merge judgment, not a deterministic action, so it belongs in the guide/skill, not the Makefile.
- **Kept deliverables land under `.claude/` by default** (e.g. `.claude/requirements/`), beside `memory/` and `skills/` — a recommended-not-imposed convention; the guide requires one versioned owner referenced by the merged `CLAUDE.md`, and the owner may repath to match project conventions. The charter and adopted modules fold into `CLAUDE.md` (no separate files); only standalone reference docs (requirements) need a home.

Both deliverables were edited in lockstep so they never drift (the skill mirrors the guide verbatim, per the 2026-07-11 skill decision); `CHANGELOG.md`, `README.md`, and `docs/index.html` refreshed in the same change. Rejected: a leaner/ephemeral adoption installer (diverges `make adopt` from `make new` for a cleanup the guide can own); leaving the kit inert or gitignored (the option orderboard declined — a lingering second copy); top-level `./requirements/` as the default home (mixes adopted docs into the project's code tree).

## 2026-07-11 — Installer (`make new`/`make adopt`), deterministic assembler, self-adoption

Owner asked for a make-style installer taking the target path. Q&A round settled four choices: (1) **assembly became a script** — `tools/assemble.py` (python3, stdlib) implements the templating and reproduces the shipped charters byte-for-byte; it is the single engine behind `make assemble`, install-time module composition, the `assemble-charters` skill, and the hook's rule 1, which upgraded from staging-presence to **regenerate-and-byte-compare**. (2) Files land in a **namespaced folder, default `agent/`**, `PREFIX=`-overridable — the owner flagged the clash risk of `docs/` mid-round, superseding the initial `docs/` answer. (3) `make new` seeds the **complete kit** (stub `CLAUDE.md` from the new `templates/CLAUDE_MD.template.md` deliverable, memory, inbox, `graduate-idea`); never overwrites; aborts to `make adopt` when a `CLAUDE.md` exists. (4) **Self-adoption is sanctioned**: `make adopt DEST=.` detects this repo and installs skill *working copies* into `.claude/skills/` (the virtuous cycle — the meta-repo is its own adopter #1); the anti-contamination rule now distinguishes authoring (only `templates/skills/`) from installed copies (refresh by re-running the installer, never hand-edit). Boundary kept: the installer is deterministic file installation only — no interactivity, no orchestration — so the parked lifecycle CLI (`ideas/lifecycle-cli.md`) stays parked. Rejected: pre-composed-only installer (loses module composition at install time); shell-only assembler (regex-heavy text surgery in POSIX sh); installing a reference set on self-adoption (duplicates `templates/`).

Rejected: keeping the marker+skill pipeline (machinery for a delta the session already knows).
- **`REQUIREMENT_PROJECT_CLI.md` parked** back to `ideas/lifecycle-cli.md` (`deferred`), its Q&A resolutions preserved there. A normative spec for software no adopter asked to build was speculation; it reactivates when a real project wants the tool. Charter references to the CLI removed.
- **Charter core slimmed to 12/13 sections; opinionated practices became opt-in modules.** The Q&A-round method is defined once inside Method; *Idea inbox* + *Spec-driven work* merged into *Ideas & specs*; memory/roadmap/decisions merged; Setup keeps parameter confirmation + minimum runnable scaffolding. New add-on modules: `MODULE_PRODUCT_AUDIENCE` and `MODULE_LIVING_DOCS`; shipped composed charters are minimal. **This reverses *Products are multi-tenant from v1* (2026-07-10)** as a universal default — the practice's full text lives in `MODULE_PRODUCT_AUDIENCE`, applied when a project is a product for an audience, not imposed on every internal tool. The extraction charter keeps a tenancy-neutral *Classify on extraction* rule (domain | instance-config | workaround) so the valuable half survives in the minimal charter.
- **`templates/README.md` (catalog) merged into the root `README.md`** — one adopter-facing page instead of a landing that half-duplicated a catalog. Reverses the "distinct catalog" split (2026-07-10 living-landing decision) now that the tree is small enough for one page. Hook simplified accordingly (README required whenever anything under `templates/` changes — also closes the recorded rule-4 gap for `templates/skills/`).
- **Incorporated `ideas/` specs deleted** (10 files; git history keeps them) — a spec whose content lives in template text is a second copy waiting to drift. Milestones 1–4 moved to `roadmap-archive.md` (the charters' own strategic-archiving practice, finally applied here).
- **New direction: validation before growth.** The roadmap's next milestone is adopting the templates in a real project and iterating from real friction — no further meta-refinement until then.

Rejected: single-tenant v1 with parameterized config (cheaper, but pays the migration later); blocking setup question with no default (friction on every project).

## 2026-07-12 — Upgrade ergonomics: `make upgrade` reads the stamp, dates the unstamped, and installs nothing

Graduated the two frictions the first real upgrade produced (orderboard) into `ideas/upgrade-ergonomics.md`. Both are about the *mechanics* of the Upgrade branch, not its judgment, and both land in one new installer target.

**Rejected:** keeping the push-kit (the merging session then lives inside the project that knows itself best — genuine, but the last three adoptions were all driven from here anyway, and it is the kit that produced two of the four frictions); deleting `tools/install.sh` and letting the skill do file operations by hand (smaller surface, but it throws away the one deterministic, end-to-end-exercised piece of machinery in the repo); documenting both doors (the owner explicitly asked to retire every other way to install); inline HTML markers in the adopter's `CLAUDE.md` (grep-able, which is literally what the note asked for — but they claim ownership of text the project is invited to rewrite, and they pollute the file the agent reads every session).

Consequence to face during incorporation: a host-side session **does not know the target project** the way its own session does. The inventory step and plan-mode approval carry that weight now, and the spec names it as the risk to watch in the first host-side adoption.
