claude·templates working agreements for AI coding agents
A charter, not a blank page

Give a coding agent a method to follow — before it writes a line.

Reusable working agreements that bootstrap Claude-driven software projects. Copy one document into your repo, fill in a short parameters block, and your agent has a disciplined method — phased discovery, written specs, prototypes, tested delivery — instead of improvising. The product here is prompt text: two minimal charters, opt-in add-on modules, a requirement, a guide, and embeddable skills — no application and no runtime; installing means pointing a script at your project's folder.

01

Install

Clone this repo and point the installer at your project's folder — that is the whole install. Files land in a namespaced agent/ folder, and nothing is ever overwritten. 

git clone https://github.com/diegopego/claude-templates
cd claude-templates

make adopt DEST=~/path/to/your-project                    # existing project
make new   DEST=~/path/to/new-project CHARTER=greenfield  # or CHARTER=legacy

Then open Claude Code in your project and paste the prompt the installer just printed:

existing project

Paste Adopt the templates in agent/ into this project. — a non-destructive merge: the agent inventories your current instructions, keeps or adapts each section per the adoption guide, and raises every conflict to you instead of overwriting. Code and infrastructure changes are deferred to a roadmap, so the running system is never touched by surprise. When the merge is done it tidies up: each section you keep lands in a single versioned home (under .claude/ by default) and the one-shot delivery kit and skill are removed.

new project

Paste Read CLAUDE.md and start Setup. — the charter's Setup step confirms every parameter with you (stated defaults, never silent) and scaffolds a minimum runnable project, so real work starts against something that already runs.

Already have a Claude Code session open in the project? Run /reload-skills there first — skills load at startup. No make? Copying the files by hand works exactly the same; the installer is convenience, not machinery.

02

What you get

Greenfield charter charter

Building a new application from scratch: interrogate the vision, turn it into written specs, prototype, then build tested slices. The risk it fights is building the wrong thing confidently.

Legacy transformation charter charter

Extract the knowledge inside an existing system — a spreadsheet, an old app, a paper process — and rebuild it as a modern application, with traceable extraction, every rule classified as domain, instance-config, or workaround, and data migration with a planned cutover.

Add-on modules modules

The shipped charters are deliberately minimal; heavier opinions are modules you compose in only when your project is that kind of project.

MODULE_PRODUCT_AUDIENCE — a product for an audience: multi-tenant from v1, your organization as tenant #1, domain rules separated from instance configuration.
MODULE_LIVING_DOCS — curated changelogs, a living product doc, and a landing page whose skin is designed once at Setup, with an approval loop on every publish; automated by the update-living-docs skill (rebuild + update).
MODULE_DATA_MIGRATION — built into the legacy charter; addable to a greenfield project that imports inherited data.

Portable-appliance requirement requirement

Every application must be destroyable and rebuildable on a fresh machine from repo + secrets + backup in under 30 minutes. The server is disposable; the backup is the system.

Adoption guide guide

Bring these practices into a project that already exists and works — without losing the instructions it already has. A non-destructive merge that raises every conflict to you.

Embeddable skills skill

Drop-in Claude Code skills a project copies into its own .claude/skills/. Three so far: graduate-idea, which drives a rough idea from the inbox through its Q&A round into an agreed spec; adopt-template, which runs the adoption guide for you — inventorying an existing project's instructions, classifying each section, and producing a merged CLAUDE.md with every conflict raised to you; and update-living-docs, the executable arm of MODULE_LIVING_DOCS — it builds and keeps a project's dual changelogs, living product doc, and landing page current, with a rebuild mode that regenerates the whole set from a project's current state (a large project adopting the pipeline mid-life, or a forced re-creation) and an update mode for single changes, publishing only behind an approval loop.

03

The method a charter runs

Work advances through explicit phases; the agent always states which one it is in.

0

Setup

Confirm every project parameter with you — languages, users, stack, all stated defaults, never silent — then scaffold a minimum runnable project: skeleton, config, a CLAUDE.md wired to the charter, seeded memory. No blank folder.

1

Discover / Understand

Interrogate the vision, or map the legacy system's structure and embedded rules. Draft a domain model.

2

Align

Q&A rounds to resolve ambiguities and rank what matters. The smallest product that delivers the vision wins.

3

Specify / Golden standards

Record the agreed rules as written specs with acceptance criteria. These become the test oracles.

4

Prototype

Build UI/UX prototypes; the align–specify–prototype loop runs until the model and prototypes are approved.

5

Build

Implement the thinnest end-to-end slice first, validated against the specs, until v1 is demonstrated by tests.

04

What the charters give a project

  • ▸Q&A rounds as the engine between phases. Related questions asked together, each with options and a recommendation, a scripted baseline then adaptive follow-ups — every answer written down as a spec or a decision. Defined once, in the Method, so the agent asks well instead of guessing.
  • ▸Setup that scaffolds. Setup settles your stack and language and generates a minimum runnable project; the charter's tech picks are recommended, overridable defaults you confirm — never silent assumptions.
  • ▸Modern strict TypeScript as the recommended default — the type system used to encode business rules.
  • ▸Anti-over-engineering. Boring solutions, few dependencies, seams not scaffolding.
  • ▸Functional core, imperative shell with TDD; the agreed specs are the test oracles.
  • ▸Versioned in-repo memory. Roadmap and decision log live in the repo, so a fresh clone can resume the work.
  • ▸Ideas & specs. A scratchpad inbox graduates half-formed ideas into proper specs through a Q&A round — the inbox proposes scope, the roadmap accepts it — and every non-trivial task becomes a short spec before it is built. The graduate-idea skill automates the ritual.
  • ▸Explicit git authorization. The agent prepares work but never commits or pushes on its own.

The heavier opinions — multi-tenancy from v1, dual changelogs, a designed landing page — are opt-in modules, not defaults: you add them when your project is that kind of project.
